## 배경

BERT 기반 NER(Named Entity Recognition) 모델을 특허 문서에 적용하면서, **BERT의 512 토큰 입력 한계**가 반복적으로 문제를 일으켰다. 특허 문서는 수만~수십만 자에 달하므로 텍스트를 청크 단위로 분할하여 BERT에 입력해야 하는데, 분할 전략에 따라 NER 결과가 완전히 달라지거나 전체가 실패하는 상황이 발생했다.

이 글에서는 청크 분할 방식을 3단계에 걸쳐 개선한 과정과, BERT 512 한계를 근본적으로 해결하기 위해 적용한 2중 방어 전략을 정리한다.

## 1. 문자 수 기반 분할의 함정

### 초기 상태: 1000자 + `. ` 구분자

초기 구현은 텍스트를 1000자 단위로 분할하되, 마침표(`. `)를 기준으로 경계를 맞추는 방식이었다.

```python
CHUNK_SIZE = 1000
dot = normalized.find(". ", search_from)
if dot == -1:
    chunk_end = len(normalized)  # 마침표가 없으면 텍스트 끝까지
```

**문제:** 특허 문서에는 마침표 없이 수천 자가 이어지는 화학식 테이블이 빈번하다. 이 경우 구분자를 찾지 못해 청크가 텍스트 끝까지 이어지면서, 최대 38,739자짜리 청크가 생성되었다. BERT 512 토큰을 대폭 초과하여 `RuntimeError: The size of tensor a (529) must match the size of tensor b (512)` 발생.

### 개선: 500/800자 다단계 분할

목표 크기를 500자로 줄이고, **하드 리밋 800자**를 도입하여 최대 청크 크기를 보장했다. 구분자도 `. ` → `; ` → `, ` → ` ` 순서로 다단계 폴백을 적용하여, 마침표가 없는 구간에서도 안전하게 분할되도록 했다.

| 문서 | 개선 전 (1000자) | 개선 후 (500/800자) |
|---|---|---|
| us010000482b2 (340K자) | 74청크, max 38,739자 → **전체 실패 (0 스팬)** | 696청크, max 478토큰 → 6,032 스팬 |
| us20220403229a1 (54K자) | 47청크, max 3,167자 → **전체 실패 (0 스팬)** | 104청크, max 364토큰 → 660 스팬 |
| us012486447b2 (36K자) | 33청크 → 319 스팬 | 66청크 → 307 스팬 |

**핵심:** BERT에 입력되는 토큰 수를 보장하지 않는 문자 수 기반 분할은 본질적으로 불안전하다. 하드 리밋과 다단계 구분자 폴백이 필수.

## 2. 토큰 기반 분할로의 전환

### 문자 수 기반의 근본적 한계

500/800자 다단계 분할로 안정성은 확보했지만, 문자 수와 토큰 수의 비율은 텍스트 특성에 따라 크게 변동한다.

- 일반 영문: 1토큰 ≈ 4~5자
- 화학식 (`2-(4-nitrobenzoyl)oxazole`): 26자에 10+ 토큰

이 때문에 500자 청크의 실제 토큰 수는 256~510으로 편차가 컸고, 토큰이 적은 청크가 많아질수록 GPU 배치 수가 불필요하게 증가했다.

### 개선: BERT 토크나이저 직접 활용

`_build_chunks_token()` 함수를 도입하여 BERT 토크나이저로 직접 토큰 수를 측정했다. `MAX_BERT_TOKENS=510`으로 각 청크가 BERT 한계를 초과하지 않도록 보장하면서, 이전 청크의 토큰 밀도(chars/token)를 **지수 이동 평균**으로 추적하여 초기 분할 위치를 추정하고, 초과 시 **이진 탐색**으로 안전한 위치를 찾는 알고리즘을 적용했다.

| 문서 | 문자 기반 (500/800자) | 토큰 기반 (510) | 청크 수 감소 |
|---|---|---|---|
| us010000482b2 | 696청크 (max 478토큰) | 399청크 (max 510토큰) | **-43%** |
| us012486447b2 | 66청크 (max 364토큰) | 30청크 (max 498토큰) | **-55%** |
| us14104208a1 | 140청크 (max 256토큰) | 55청크 (max 509토큰) | **-61%** |

청크 수가 43~61% 감소하여 GPU 배치 오버헤드가 대폭 줄어들었다.

### 그러나 — 외부 토크나이저 ≠ 내부 토크나이저

`MAX_BERT_TOKENS=510`으로 설정했음에도 다시 RuntimeError가 발생했다.

```
RuntimeError: The size of tensor a (513) must match the size of tensor b (512)
```

**근본 원인은 토큰화 경로의 불일치:**

```
청크 분할 경로:  _bert_tokenizer.encode(raw_text)  → 직접 서브워드화 (1단계)
CDE 내부 경로:   raw_text → CDE word tokenizer → BERT subword → input_ids (2단계)
```

두 경로는 공백/구두점/화학식 처리 방식이 달라 동일 텍스트에서 서로 다른 토큰 수를 산출한다. 청크를 510 토큰으로 제한해도 CDE 내부에서 513 토큰을 생성하는 사례가 확인되었다.

**교훈:** 외부 토크나이저로 측정한 토큰 수와 라이브러리 내부의 토큰 수가 일치한다는 가정은 위험하다. 토큰 기반 분할은 청크 수를 50% 이상 줄이는 확실한 효과가 있지만, 한계값에 여유분이 필요하다.

## 3. 2중 방어: 여유분 + Position Embedding 확장

### truncation이 불가능한 이유

직관적으로는 "512를 초과하면 잘라내면 되지 않나?"라고 생각할 수 있지만, CDE(ChemDataExtractor)처럼 BERT 출력의 모든 토큰을 후속 파이프라인에서 소비하는 구조에서는 truncation이 파이프라인 전체의 차원 일관성을 깨뜨린다.

| 시도 | 접근 | 실패 원인 |
|---|---|---|
| 1차 | `bert_model.forward`에서 `input_ids`만 truncation | CRF의 mask/offsets 텐서와 차원 불일치 → CUDA assert |
| 2차 | `model.forward`에서 모든 텐서 truncation + offsets clamp | CDE가 단어 수와 태그 수 일치를 검증 → TypeError |
| 3차 | offsets clamp만 유지, crf_mask 변경 제거 | `_assign_tags`에서 225단어 vs 224태그 불일치 |

CDE는 `BERT → CRF → 단어-태그 매핑 → 문장-태그 할당`의 파이프라인에서 각 단계가 이전 단계의 출력 차원에 의존한다. 한 곳을 자르면 연쇄적으로 차원 불일치가 전파된다.

### 최종 해결: "자르지 않고 수용"

**1차 방어 — MAX_BERT_TOKENS: 510 → 506**

CDE 내부 토큰화 오차(대부분 4토큰 이내)를 흡수하는 여유분을 확보했다.

**2차 방어 — Position Embedding 테이블 확장 (512 → 1024)**

BERT 모델의 position embedding을 런타임에 확장하여, 512를 초과하는 입력이 들어와도 RuntimeError 없이 처리되도록 했다.

```python
_NEW_MAX_POS = 1024
old_emb = bert.embeddings.position_embeddings        # (512, hidden)
new_emb = torch.nn.Embedding(_NEW_MAX_POS, old_emb.embedding_dim)
new_emb.weight[:old_max_pos] = old_emb.weight         # 0~511: 학습된 값 유지
new_emb.weight[old_max_pos:] = old_emb.weight[-1]     # 512~1023: 마지막 위치 복제
```

위치 512+ 토큰은 모두 동일한 embedding(위치 511)을 사용하므로 해당 구간에서 위치 구분 능력이 상실되지만, 1차 방어(MAX=506)가 대부분의 경우를 흡수하므로 실제 512를 초과하는 토큰은 0~14개 수준이다.

### 검증 결과

| 문서 | Before (510, 미확장) | After (506 + pos_embed 1024) |
|---|---|---|
| us010000482b2 | **0 스팬 (전체 실패)** | 6,036 스팬 |
| us012486447b2 | 310 스팬 | 310 스팬 (100% 일치) |
| us14104208a1 | 627 스팬 | 631 스팬 (Jaccard 97.18%) |
| us20220403229a1 | 664 스팬 | 665 스팬 (Jaccard 99.85%) |

## 4. CDE batch_tag의 치명적 전파 특성

이 문제가 특히 위험한 이유는 CDE의 `_batch_assign_tags` 설계에 있다. Document 내 전체 문장을 하나의 BERT forward 배치로 처리하기 때문에, **단 1개 문장이라도 512 토큰을 초과**하면 해당 배치 전체가 RuntimeError를 발생시키고, Document에 속한 **모든 청크의 NER 결과가 소실**된다.

- us010000482b2 (Before): 399개 청크 중 1개 문장이 513 토큰 → 399개 전부 실패 → 0 스팬, 7분 낭비
- us010000482b2 (Baseline): 74개 청크 중 다수가 529+ 토큰 → 74개 전부 실패 → 0 스팬, 2.8분 낭비

"대부분의 경우 괜찮다"가 아니라 **"단 한 번이라도 초과하면 전체가 무너진다"**는 것이 핵심이다. 2중 방어(토큰 여유분 + position embedding 확장)가 과잉 방어가 아닌 이유다.

## 5. BIO 태그 기반 CEM 추출로의 전환

청크 분할과 별도로, NER 결과에서 화학 엔티티(CEM)를 추출하는 방식도 개선했다.

### 기존 방식: 재귀 하강 파서

CDE 기본 `chemical_name.scan()` 파서는 재귀 하강 방식으로 토큰당 ~35회 함수 호출, O(n·k) 복잡도를 가진다. 프로파일링 결과 CEM 추출이 전체 NER 시간의 ~85%를 차지했다 (600문장 기준 2.6초). 또한 파서가 `benzothiazoles/thiazoles` 형태의 복합명을 하나의 엔티티로 합치는 품질 문제도 있었다.

### 개선: BIO 태그 선형 스캔

`Sentence.cems`를 monkey-patch하여 BERT-CRF가 생성한 BIO 태그(`B-CM`/`I-CM`) 기반 O(n) 선형 스캔으로 교체했다. 후처리(IGNORE_PREFIX/SUFFIX, 괄호 제거, SPLITS, SPECIALS)는 유지하되, 개별 endswith/startswith 루프를 단일 정규식으로 교체하여 추가 최적화했다.

| 문서 | 원본 파서 (재귀 하강) | Fast CEM (BIO 스캔) | Jaccard |
|---|---|---|---|
| us012486447b2 | 319 스팬 | 307 스팬 | 94.41% |
| us14104208a1 | 630 스팬 | 627 스팬 | 93.09% |

차이 패턴:
- 원본 파서: `"ferrocene/ferrocenium"`, `"DMSO/THF"` — 슬래시 복합명을 하나로 합침
- BIO 스캔: `"ferrocene"` + `"ferrocenium"` — 개별 화학물질로 올바르게 분리

**핵심:** 파서가 BIO 태그를 무시하고 독자적으로 스팬을 결정하면, NER 모델의 예측과 불일치가 발생한다. BIO 태그를 직접 활용하는 것이 모델 의도에 부합하고, 복합명 분리도 더 정확하며, 성능도 O(n·k) → O(n)으로 대폭 개선된다.

Big-O 표기상으로는 k≈35가 상수이므로 둘 다 O(n)이지만, 실측에서 85%가 단축된 이유는 그 상수의 내용물이 **Python 함수 호출**이기 때문이다. Python은 함수 호출마다 스택 프레임 객체의 힙 할당, 인자 바인딩, 지역 변수 딕셔너리 초기화가 수반되므로 호출 비용이 크다. 토큰당 35회 함수 호출이 토큰당 1회 문자열 비교로 바뀌면, 상수 팩터만으로 수십 배의 실질적 차이가 발생한다.

## 전체 변경 이력과 NER 품질 영향

```
pre-b4f662f ──── b4f662f ──── 02d5e90 ──── 6d0e865 ──── 900ea9d
 (Baseline)      (Legacy)                   (Before)      (After)
```

| 상태 | 청크 방식 | CEM 추출 | 512 방어 | 4개 샘플 스팬 합계 |
|---|---|---|---|---|
| Baseline | 1000자, `. `만 | 원본 파서 | 없음 | **949** (2건 전체 실패) |
| Legacy | 500/800자, 다단계 | Fast BIO | 없음 | **7,626** |
| Before | 토큰 510 | Fast BIO | 불완전 | **1,601** (1건 전체 실패) |
| After | 토큰 506 + pos_embed 1024 | Fast BIO | 2중 방어 | **7,642** |

변경이 누적될수록 NER 결과의 차이가 벌어지지만, 인접 단계 간 Jaccard는 95% 이상으로 안정적이다. 차이의 대부분은 청크 경계 이동에 의한 문맥 변화로, 실질적 정보 손실이 아니다.

## 정리

| 교훈 | 설명 |
|---|---|
| 문자 수 ≠ 토큰 수 | 화학 텍스트에서는 문자당 토큰 비율이 일반 텍스트의 2배 이상 |
| 외부 토크나이저 ≠ 내부 토크나이저 | 라이브러리가 자체 전처리를 거치면 동일 텍스트에서도 토큰 수가 다름 |
| Truncation < 수용 | 후속 파이프라인이 BERT 출력 전체를 소비하는 구조에서는 자르는 것보다 수용하는 것이 안전 |
| 1건의 실패 = 전체의 실패 | 배치 처리 구조에서 1건의 512 초과가 전체 배치를 무너뜨림 |
| 모델 예측을 존중 | NER 모델이 생성한 BIO 태그를 직접 활용하는 것이 별도 파서보다 정확하고 빠름 |
