## 배경

단일 GPU(A100)에서 BERT 기반 NER API를 운영하면서, 다수 요청이 동시에 들어올 때 처리 시간이 급증하는 병목을 겪었다. 순차 단건 1초 이내였던 처리가 동시 요청 시 10초 이상으로 악화되었고, 이를 해결하기 위해 세마포어 기반 동시성 제어, 입력 크기 제한, 그리고 요청 배칭까지 단계적으로 적용한 과정을 정리한다.

---

## 1. CUDA 커널 경합과 Semaphore

### 문제: 무제한 GPU 접근

GPU 접근에 대한 동시성 제어가 없어 모든 요청이 무제한으로 GPU에 접근했다. 다수 요청이 동시에 CUDA 커널을 실행하면 경합(contention)이 발생하여 개별 요청 처리 시간이 급증한다.

### 해결: asyncio.Semaphore

`asyncio.Semaphore(NER_GPU_CONCURRENCY)`를 도입하여 모든 NER 엔드포인트에서 GPU 추론 구간만 세마포어로 보호했다. CPU 작업은 세마포어 밖에서 실행하여 대기 중에도 전처리가 가능하도록 설계했고, 응답에 `queue_ms` 필드를 추가하여 GPU 대기 시간을 가시화했다.

```
10건 순차 기준 건당 중앙값: 8.9s → 0.8s (91% 단축)
```

### Semaphore 값의 트레이드오프

세마포어 값은 단순히 "높을수록 좋다"가 아니다.

- **Semaphore(1):** 경합이 완전히 제거되지만 GPU 유휴 시간이 발생. 소형 문서가 GPU를 10~20%만 사용해도 나머지는 놀게 됨.
- **Semaphore(3):** GPU 활용률이 올라가지만 경합 위험도 함께 증가. 건당 체감 시간이 3~5초로 늘어남.

또한 wall time(총 소요 시간)과 건당 체감 시간(큐 대기 포함)을 구분해서 해석해야 한다. wall time이 동일해도 동시 요청 시 건당 체감 시간은 큐잉 때문에 길어진다.

## 2. 입력 크기 제한

### 문제: 극단적 처리 시간 편차

특허 문서(title + abstract + description + claims)가 수만~수십만 자에 달해 수백 개의 청크가 BERT-CRF를 통과했다. 50건 기준 평균 41.8초 vs 중앙값 9.6초로 편차가 극심했다.

처리 시간이 문서 크기에 비례하는 이유는 **텍스트 길이 → 청크 수 → BERT forward 횟수**의 관계 때문이다. 340,000자 문서는 ~696개 청크를 생성하고, 각 청크가 BERT forward를 1회씩 통과해야 한다. 대형 문서 한 건이 GPU를 수분간 독점하면서 전체 서비스의 응답 시간이 급증했다.

### 해결: 텍스트 절삭

`NER_MAX_TEXT_LENGTH`(기본 100,000자) 초과 시 문장 경계(`. `)에서 절삭하는 `_truncate_text()` 함수를 도입했다. 응답에 `truncated` 플래그를 추가하여 호출자가 절삭 여부를 인지할 수 있게 했다.

절삭의 효과는 문서 자체를 빠르게 처리하는 것이 아니라, **BERT를 통과하는 청크 수의 상한선을 설정**하는 것이다. 100,000자로 제한하면 최대 ~200개 청크로 캡이 걸려, 대형 문서가 GPU를 무한정 점유하는 상황을 방지한다.

```
세마포어 + 절삭 적용 후 50건 순차 기준 건당 평균: 41.8s → 0.8s
```

단, 이 수치는 세마포어(1절)와 절삭이 모두 적용된 복합 결과다. 절삭 단독의 기여는 극단적으로 큰 문서의 처리 시간 상한선을 낮추는 것이며, 일반적인 크기의 문서에는 영향이 없다.

**핵심:** NER 결과의 완전성보다 서비스 안정성이 우선인 경우, 입력 크기를 제한하는 것이 가장 효과적인 방어 수단이다. 다만 절삭 값을 너무 낮게 설정하면 대형 문서 후반부의 엔티티가 누락되므로, 서비스 요구사항에 맞춰 균형점을 찾아야 한다.

## 3. Request Batching: 독립 실행 vs 일괄 처리

### Semaphore 방식의 한계

세마포어 방식에서는 각 요청이 독립적으로 BERT forward를 실행한다. 소형 문서 1건이 GPU를 10~20%만 사용해도 나머지 CUDA 코어는 유휴 상태가 된다.

### Batching: 모아서 한 번에

백그라운드 `_batch_worker` 코루틴이 `asyncio.Queue`에서 요청을 수집한다. 첫 아이템 도착 후 `NER_BATCH_MAX_WAIT_MS`(50ms)까지 대기하거나 `NER_BATCH_MAX_SIZE`(8)에 도달하면 즉시 `extract_chemical_spans_multi`로 단일 GPU 배치 처리한다. Document 생성 1회, BERT forward 1회로 다수 문서를 처리한다.

```
50건 배치 기준: 배치 결과들이 8건 단위로 동시에 반환되는 패턴 확인
GPU 활용률이 높아져 동일 wall time에 더 많은 요청 처리 가능
```

### 두 접근법의 근본적 차이

| | Semaphore | Batching |
|---|---|---|
| **원리** | 독립 요청 N개가 GPU를 동시 사용 | 요청 N개를 모아 GPU 1회 호출 |
| **GPU 호출** | 요청 수만큼 별도 forward | 배치 1회의 forward |
| **CUDA 코어 활용** | 요청당 10~20% (소형 문서) | batch_size에 비례하여 증가 |
| **추가 지연** | 없음 | 수집 대기 시간 (50ms) |
| **적합한 상황** | 단건 요청이 간헐적으로 오는 경우 | 동시 트래픽이 많은 경우 |

GPU는 행렬 연산 장치이므로 `batch_size`가 클수록 CUDA 코어 활용률이 비례하여 올라간다. 그러나 배칭에는 수집 대기 시간(50ms)이 추가되므로, 단건 요청이 드문드문 오는 상황에서는 오히려 지연이 발생할 수 있다.

`NER_BATCH_MODE` 환경변수로 기존 세마포어 방식과 전환 가능하도록 설계하여, 트래픽 패턴에 따라 선택할 수 있게 했다.

## 정리

| 단계 | 기법 | 효과 |
|---|---|---|
| 동시성 제어 | `asyncio.Semaphore` | CUDA 경합 제거 |
| 입력 크기 제한 | `_truncate_text()` | 처리 시간 상한선 설정 |
| 요청 배칭 | `asyncio.Queue` + batch worker | GPU 활용률 극대화, 동시 요청 처리량 증가 |

GPU 최적화는 단일 기법이 아니라 계층별로 적용해야 효과적이다. 하드웨어(GPU 경합 방지) → 입력(크기 제한) → 스케줄링(배칭) 순서로 병목을 제거하면 각 단계의 효과가 누적된다.
