## 배경

NER API의 GPU 병목을 개선하면서 반복적으로 마주친 질문이 있다. "GPU 활용률을 더 올릴 수 있지 않나?" 올릴 수 있다. 하지만 공짜가 아니다. 이 글에서는 단일 GPU 서비스에서 **활용률, 건당 체감 지연, 안정성** 세 축의 트레이드오프를 정리하고, 서비스 특성에 따른 선택 기준을 논의한다.

---

## 세 축의 트레이드오프

```
                높은 GPU 활용률
                      ▲
                     / \
                    /   \
                   /     \
  낮은 건당 지연 ◄─────────► 높은 안정성
```

세 가지를 동시에 최대화할 수 없고, 하나를 올리면 다른 하나가 내려간다.

---

## 축 1: GPU 활용률 ↔ 건당 체감 지연

|  | GPU 활용률 낮음 | GPU 활용률 높음 |
|---|---|---|
| **방법** | Semaphore(1) — 1건씩 직렬 | Batching 8건 또는 Semaphore(3) |
| **GPU 사용** | 10~20% (소형 문서 기준) | 60~80% |
| **건당 체감 시간** | 순수 처리 시간만 (0.4~1.2s) | 수집 대기 + 큐 대기 추가 |
| **단건 요청** | 즉시 처리 | 배칭 대기 50ms 추가 |

GPU를 꽉 채우려면 여러 요청을 모아야 하고, 모으는 동안 먼저 도착한 요청은 기다려야 한다. **동시 트래픽이 많으면 배칭이 유리**하고, **단건이 간헐적으로 오면 직렬이 유리**하다.

---

## 축 2: GPU 활용률 ↔ 안정성

|  | 보수적 | 공격적 |
|---|---|---|
| **방법** | Semaphore(1), 텍스트 절삭 | Semaphore(5+), 절삭 없음 |
| **GPU 메모리** | 여유 있음 | VRAM 피크 시 OOM 위험 |
| **처리 예측성** | 건당 시간 일정 | 문서 크기에 따라 편차 극심 |
| **장애 확률** | 거의 없음 | CUDA OOM → 프로세스 크래시 |

A100이 80GB VRAM이라 여유가 있지만, 대형 문서 여러 건이 동시에 GPU에 올라가면 메모리가 폭증할 수 있다. 활용률을 올리면 그만큼 피크 메모리가 예측 불가해진다.

---

## 축 3: 건당 체감 지연 ↔ 안정성 (Latency vs Throughput)

|  | Latency 최적화 | Throughput 최적화 |
|---|---|---|
| **목표** | 개별 사용자의 응답 시간 최소화 | 단위 시간당 최대 처리량 |
| **전략** | Semaphore(1), 도착 즉시 처리 | Batching, 수집 후 일괄 처리 |
| **Throughput** | ~0.7s/건 (변하지 않음) | GPU 배치 효율로 ~0.5s/건 가능 |
| **P99 Latency** | 안정적 | 수집 대기로 꼬리 지연 발생 가능 |

---

## 설정별 위치

```
Semaphore(1)          Semaphore(3)          Batching(8)
안정 ◄──────────────── 현재 ─────────────────► 효율
     GPU 20%               GPU 50%               GPU 70~80%
     건당 0.8s              건당 3~5s(동시)         건당 가변
     장애 거의 없음         경합 소폭               수집 대기 추가
```

`NER_GPU_CONCURRENCY=3` + `NER_BATCH_MODE=true` 설정은 효율 쪽으로 꽤 치우쳐 있는 상태다. 배칭이 켜져 있으니 세마포어 3은 사실상 사용되지 않고, 배치 워커가 8건씩 모아 GPU를 호출한다.


---

## 선택 기준

어떤 축을 우선할지는 서비스 특성에 따라 결정해야 한다.

### 실시간 사용자 응답이 중요한 경우 → Latency 우선

- Semaphore(1)로 즉시 처리
- 텍스트 절삭으로 처리 시간 상한선 보장
- GPU 활용률은 포기 (10~20%)

### 대량 배치 처리가 주 용도인 경우 → Throughput 우선

- Batching으로 GPU 활용률 극대화
- 수집 대기 시간은 감수
- 건당 체감 시간보다 전체 wall time 최적화

### 장애가 절대 발생하면 안 되는 경우 → 안정성 우선

- Semaphore(1) + 텍스트 절삭 + 보수적 설정
- GPU 활용률과 throughput 모두 포기
- 건당 시간 일정, 장애 확률 최소화

---

## 정리

A100 한 대의 활용률을 올리는 것은 가능하지만, 공짜가 아니다.

- **활용률 ↑** = 건당 체감 지연 ↑ (큐/수집 대기)
- **활용률 ↑** = 안정성 ↓ (VRAM 피크, 예측 불가)
- **건당 지연 ↓** = 활용률 ↓ (GPU가 놀게 됨)

기술적 최적화에서 중요한 것은 "어떻게 올리느냐"보다 **"무엇을 포기할 수 있느냐"**를 먼저 정하는 것이다. 트레이드오프의 존재를 인식하고, 서비스의 우선순위에 따라 의식적으로 선택하는 것이 올바른 엔지니어링 판단이다.
